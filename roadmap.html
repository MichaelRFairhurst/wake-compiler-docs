Wake Development Roadmap

There are many interdependent features, and high target goals. The relationships are expressed here, each in their own section, with high importance items bordered in darker blue.


1. Type Tags

Type tags might prove to be a flagship feature of Wake, but to do them right and/or get their whole benefit involves several moving parts.

ready to develop

Parameterized Ranges (#20)

Declaring generic classes with restrictions on their parameterized types, ie, Cone{T from Icecream}.

Incredibly useful for generic programming, and should be available for std lib development/design.

Self Type

A special generic a type that represents this. Extending the class, self becomes the extension.

ready to develop

Type Tags (#37)

A freeform typechecked label for distinguishing types with the compiler's help. Example: keeping Num:miles and Num:kilometers apart.

Typesafe Autoboxing

The standard library will now be able to have methods such as Self -- maxOf(Self)

end

ready to develop

Service Extensions (#78)

Classes that add methods to other classes when in scope. This is nice for extending the std library, or other libraries, and can lead to cleaner tests to boot.

Operator overloading

Instead of adding methods to other classes, they create operator overloading. Combined with type tags, this can be used to automatically convert units on arithmatic/assignment.

2. JS Interop

There are a lot af parts in Wake, and a lot of parts in JavaScript. Getting all of them to play nice on each side is starting to wrap up, but still not done yet.

for a more detailed and up-to-date summary go to the interop plan summary.

Runtime method segment stripper (124)

runtime function to turn complex method names into simple ones (ie, printLine(Text) to printLine), aborting on unusable symbols like "that(Bool)Equals(Bool)"

Deminification runtime (125)

Runtime function to loop over object properties and deminify them, store the original wake on the js, and store/reuse the js on the original wake.

Recursive deminification (126)

Recurively wrap wake object properties (including functions ie methods), and use Object.setProperty() to do so, lazily.

Generate reminification setters (127)

Use Object.setProperty to intercept when properties on wrappers around wake objects are being set. Unwrap the value and set it on the original Wake object.

Wake to JS function conversion (128)

Handle functions sent out of wake. Unwrap wake objects on the way in, wrap return values, and save the relationship between the two for quick reversal.

Primitives into Wake (129)

Handle primitives sent into wake. Detect undefineds and turn them into nulls. Otherwise, check for attached original wake objects, if none exists assume its a primitive and leave it untouched (for now).

Arrays in&out of Wake (130)

Handle arrays: Arrays coming into wake are marked js arrays. Arrays sent out of wake must have all items transformed to js objects must be marked js arrays.

Js to Wake function conversion (131)

Unwrap each js argument sent in, wrap the return in js, and save the relationship between the two for quick reversal.

Extern Js Wrappers (132)

In wake-js-gen, have extern wrappers that contain secret ctor args. These wrappers should preset the 'wake original' and 'js representation' for usage in other components.

Create extern js wrappers automatically (133)

Create a runtime function for ahead-of-time generation of wake/js linking on certain functions.

Ability to link as a js lib (134)

rather than generate an injected MainClass.mainMethod(), we generate a Wake.Constructor(args) approach...need to think about which ctors to expose. All of them? And un-js the wake ctor arguments. Also need to make it usable with ecmascript6 modules and nodejs importing.

JS Singletons + function layers (135)

For serving plain functions to Wake and/or things which only need to be subclassed in the js world.

jswrappers + generics (136)

Generic classes may accept extern js wrappers as args at times, and not at other times. Need to find when and pass in ctors to the generic to make it handle these cases.

Async/await (116)

Add async/await support so that we can have an async library which isn't painful to use in other languages where async isn't as much of a necessity.

Angular support

I would love to support angularjs well.

jQuery support

Now with wrappers for the return value of $(selector) and .offset(), and with closures to pass into .click(), we can support jQuery. This will likely teach us some lessons about where we have holes in our interp support

Sweet website to demonstrate Wake

Take a look at our proof-of-concept interactive wake explorer, which really should be rewritten in Wake before we finish it and share it.

end

3. Closures

Closures are blocked by proper type anlaysis, and then we have a couple routes to start delivering decent anonymous functions which should likely preceed closures.

 

complete

Decide Anonymous Function Syntax (#83)

The ability to annotate provisions, so that you can annotate them with a javascript method name.

complete

Code gen and semantics for anonymous functions

Implement the parsing, then the semantics, then the code gen in that order.

ready to develop

Javascript callback interop

This will enable us to interface with libraries like jQuery and angular.

end

 

complete

Anonymous Function Typechecking (#84)

Proper analysis of reassigning functions to subtypes of functions and/or making arrays of subtly different lambdas are all critical to have right.

ready to develop

Provide anonymous functions

Providers currently can only provide classes and primitives, need to provide these lambdas once we can define them.

ready to develop

Decide Method Currying Syntax (#85)

Ability to partially (or not at all) apply methods to create functions.

Code gen and semantics for method currying

Implement the parsing, then the semantics, then the code gen in that order.
